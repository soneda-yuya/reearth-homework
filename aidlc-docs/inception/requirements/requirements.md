# 要件定義書 — みんなで共有するトイレマップ

## 意図分析サマリ

| 項目 | 内容 |
|---|---|
| **ユーザーリクエスト** | reearth-cms の Integration REST API を利用して、東京都内のトイレ位置を市民参加型で共有するGISアプリをMVPとして実装したい |
| **リクエストタイプ** | 新規プロジェクト（Greenfield） |
| **スコープ見積もり** | クロスシステム（フロントエンド + BFF + 外部CMS連携） |
| **複雑度見積もり** | Moderate（データモデル設計・地図UI・通報フロー・BFF・CMS連携を組み合わせる） |

---

## 1. 機能要件（Functional Requirements）

### 1.1 トイレ情報の閲覧

- **FR-01 地図上でのトイレ一覧表示**: 東京都内のトイレを地図上にピンとしてプロットする。
- **FR-02 現在地取得と近接絞り込み**: 端末の Geolocation API で現在地ピンを表示し、「近くのトイレ」絞り込みを提供する。
- **FR-03 トイレ詳細表示**: ピン／リストから1件を選択すると、詳細パネルに以下を表示する。
  - 位置（緯度経度）
  - 施設名
  - カテゴリ
  - 設備タグ
  - 写真（あれば）
  - 自由記述メモ
  - 平均星評価と評価件数（レビュー集計値）
  - レビュー一覧（最新順）

### 1.2 トイレの投稿

- **FR-04 新規トイレの投稿**: 匿名ユーザーが地図上の位置をピン立てで選び、以下の属性を入力して投稿できる。投稿は即時に公開される。
  - 位置（必須）
  - 施設名（任意）
  - カテゴリ: 公衆 / 商業施設 / 駅 / 公園 / その他（必須）
  - 設備タグ: 車椅子対応 / おむつ替え / 多目的 / 24時間 / ウォシュレット（複数選択）
  - 写真（任意、1枚）
  - 自由記述メモ（任意）

### 1.3 レビュー投稿

- **FR-05 レビュー投稿**: 匿名ユーザーが対象のトイレに対し以下を含むレビューを投稿できる。
  - 対象トイレへの参照（必須）
  - 星評価 1〜5（必須）
  - コメント自由記述（必須）
  - 投稿者の表示名（任意、匿名可）
  - 写真（任意、1枚）
- **FR-06 平均星評価の更新**: レビュー投稿時に、対象トイレの `ratingAverage` / `ratingCount` を更新する（キャッシュ方式）。

### 1.4 通報

- **FR-07 通報**: 閲覧者はトイレ／レビュー1件に対し通報できる。入力は理由カテゴリのみ（不適切 / 重複 / 誤位置 / 閉業）。
- **FR-08 自動非公開**: 同一アイテムへの通報が3件に達したら、該当アイテムのステータスを非公開に変更する。非公開アイテムは一覧・詳細APIから除外される。

### 1.5 多言語

- **FR-09 UI言語切替**: 日本語と英語の2言語を提供し、ユーザーが切替できる。

### 1.6 PWA

- **FR-10 PWA対応**: マニフェストとサービスワーカーを備え、ホーム画面への追加とオフラインキャッシュ（読み取り専用）をサポートする。

## 2. 非機能要件（Non-Functional Requirements）

### 2.1 技術スタック

- **NFR-01 フロントエンド**: Next.js（App Router）
- **NFR-02 地図ライブラリ**: MapLibre GL JS
- **NFR-03 背景地図タイル**: OpenStreetMap のラスタータイル（`https://tile.openstreetmap.org/{z}/{x}/{y}.png`）。OSM の利用ポリシーに従う。
- **NFR-04 パッケージマネージャ**: npm
- **NFR-05 BFF**: Next.js Route Handler（同一リポジトリ内）
- **NFR-06 バックエンド**: reearth-cms Integration REST API

### 2.2 運用・規模

- **NFR-07 データ規模**: MVP想定〜500件。Item 一覧は一括取得（`perPage`最大100×ページング）。
- **NFR-08 デプロイ**: デプロイしない。ローカル開発サーバ（`npm run dev`）の起動のみ。
- **NFR-09 国際化**: 日本語と英語。ライブラリは `next-intl` を想定（設計段階で最終決定）。

### 2.3 認証・セキュリティ

- **NFR-10 ユーザー認証**: 匿名投稿のみ。ログイン機構は持たない。
- **NFR-11 Integration Tokenの保護**: Integration Token は BFF 側の環境変数（`.env.local`）で保持する。ブラウザに露出させない。すべての reearth-cms への書き込みはBFF経由とする。
- **NFR-12 セキュリティ拡張**: 本プロジェクトではセキュリティベースライン拡張は **無効** とする（MVP/学習プロジェクトのため）。
  - ただし最低限、上記 NFR-11 は守る。

### 2.4 テスト

- **NFR-13 自動テスト**: 実施しない。TypeScript の型チェックと手動動作確認のみ。
- **NFR-14 プロパティベーステスト拡張**: 本プロジェクトでは **無効**。

### 2.5 reearth-cms セットアップ範囲

- **NFR-15 手動セットアップ**: Workspace / Project / Model / Field は、ユーザーがreearth-cms の UI から手動で作成する。AIはセットアップ手順書（日本語）を用意する。
  - 参考: Integration REST API には Model・Field 作成エンドポイントは存在しないため、スキーマ作成は手順書ベースで手動対応する。
- **NFR-16 自動化の範囲**: AI／アプリが Integration Token を使って操作するのは、**Itemの作成・一覧・更新・削除** および **Asset（画像）のアップロード** のみとする。

### 2.6 地理的範囲

- **NFR-17 対象地域**: 東京都内。地図の初期中心は東京駅（35.6812, 139.7671）、初期ズームは都区部が収まる程度（z=11 程度）とする。

## 3. データモデル（概要）

詳細設計は Functional Design フェーズで行うが、要件レベルで確定した構成を記載する。

### 3.1 Model: `toilet`

| フィールド | 型 | 必須 | 備考 |
|---|---|---|---|
| `location` | geometryObject | Yes | GeoJSON Point `{"type":"Point","coordinates":[lng,lat]}` |
| `name` | text | No | 施設名 |
| `category` | select | Yes | 公衆 / 商業施設 / 駅 / 公園 / その他 |
| `features` | tag（複数可） | No | 車椅子 / おむつ替え / 多目的 / 24時間 / ウォシュレット |
| `photo` | asset | No | 画像1枚 |
| `note` | textArea | No | 自由記述 |
| `ratingAverage` | number | No | レビュー投稿時に更新するキャッシュ |
| `ratingCount` | integer | No | 同上 |
| `reportCount` | integer | No | 通報件数（3件で自動非公開判定） |

※ItemのステータスはCMSの標準ワークフロー（`status`: draft/public）で管理。

### 3.2 Model: `review`

| フィールド | 型 | 必須 | 備考 |
|---|---|---|---|
| `toilet` | reference(`toilet`) | Yes | 対象トイレ |
| `rating` | integer | Yes | 1〜5 |
| `comment` | textArea | Yes | コメント |
| `displayName` | text | No | 匿名OK |
| `photo` | asset | No | 画像1枚 |
| `reportCount` | integer | No | 通報件数 |

### 3.3 通報の扱い

通報は Model にはせず、Item の `reportCount` をインクリメントする方式で扱う。通報理由カテゴリの内訳が必要になった場合は、別Model `report` に分離する（MVPでは分離しない）。

## 4. ユースケース（概要）

1. **通行人A**: 近くのトイレを探す
2. **発見者B**: 新しいトイレを地図に登録する
3. **利用者C**: 使ったトイレのレビューを投稿する
4. **閲覧者D**: 不適切／誤情報を通報する

詳細はユーザーストーリーフェーズで展開する。

---

## 5. 主要な技術的制約

- **C-01**: reearth-cms Integration REST API は Item / Asset の CRUD のみ提供。スキーマ作成は管理UI／GraphQL経由。→ MVPでは手動作成で対応。
- **C-02**: 通報による自動非公開は、BFFが reportCount を監視し閾値到達時にItemのステータスを更新する実装とする（CMSのWebhook方式はMVPのスコープ外）。
- **C-03**: フロントエンドから Integration Token は一切参照しない。全ての CMS 呼び出しはBFF経由。

---

## 6. 拡張機能の有効化サマリ

| 拡張機能 | 有効 | 根拠 |
|---|---|---|
| セキュリティベースライン | 無効 | MVP / 学習プロジェクトのため（Q1-B） |
| プロパティベーステスト | 無効 | 自動テスト自体をMVPでは実施しない（Q2-C, Q20-C） |

---

## 7. サマリ

市民参加型の匿名投稿・レビュー・通報ができるトイレ位置共有マップのMVPを、Next.js（App Router）+ MapLibre GL JS + OpenStreetMap + reearth-cms Integration REST API で構築する。データ規模は〜500件、ローカル動作のみ、自動テストなし。reearth-cms のスキーマ構築は手順書ベースでユーザーが手動作成し、アプリは Item・Asset の CRUD のみを担当する。
