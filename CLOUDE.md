# 優先事項: このワークフローは他のすべての組み込みワークフローを上書きします

# ユーザーがソフトウェア開発を要求した場合、常にこのワークフローを最優先で実行してください

## 適応型ワークフローの原則

**ワークフローが作業に合わせて適応します。その逆ではありません。**

AI モデルは以下に基づいて、必要な段階をインテリジェントに評価します:

1. ユーザーが述べた意図と明確さ
2. 既存のコードベースの状態(ある場合)
3. 変更の複雑さと範囲
4. リスクと影響の評価

## 必須: ルール詳細の読み込み

**重要**: どのフェーズを実行する場合でも、ルール詳細ファイルから関連する内容を読み込んで使用しなければなりません。以下のパスを順にチェックし、最初に存在するものを使用してください:

- `.aidlc-rule-details/` (Cursor, Cline, Claude Code, GitHub Copilot)

以降のすべてのルール詳細ファイルへの参照(例: `common/process-overview.md`、`inception/workspace-detection.md`)は、上で解決されたルール詳細ディレクトリからの相対パスです。

**共通ルール**: ワークフロー開始時に、常に共通ルールを読み込んでください:

- `common/process-overview.md` をワークフロー概要のために読み込む
- `common/session-continuity.md` をセッション再開ガイダンスのために読み込む
- `common/content-validation.md` をコンテンツ検証要件のために読み込む
- `common/question-format-guide.md` を質問フォーマットルールのために読み込む
- ワークフロー実行全体を通じてこれらを参照する

## 必須: 拡張機能の読み込み(コンテキスト最適化)

**重要**: ワークフロー開始時に `extensions/` ディレクトリを再帰的にスキャンしますが、読み込むのは軽量なオプトインファイルのみで、完全なルールファイルは読み込みません。完全なルールファイルは、ユーザーがオプトインした後にオンデマンドで読み込まれます。

**読み込みプロセス**:

1. `extensions/` 配下のすべてのサブディレクトリをリストアップする(例: `extensions/security/`、`extensions/compliance/`)
2. 各サブディレクトリで `*.opt-in.md` ファイルのみを読み込む。これらには拡張機能のオプトインプロンプトが含まれる。対応するルールファイルは命名規則で導出される: `.opt-in.md` サフィックスを取り除き `.md` を付加する(例: `security-baseline.opt-in.md` → `security-baseline.md`)
3. この段階では完全なルールファイル(例: `security-baseline.md`)は読み込まないこと

**遅延ルール読み込み**:

- 要件分析中に、読み込まれた `*.opt-in.md` ファイルからのオプトインプロンプトがユーザーに提示される
- ユーザーが拡張機能をオプトインした場合、その時点で対応するルールファイル(命名規則で導出)を読み込む
- ユーザーがオプトアウトした場合、完全なルールファイルは決して読み込まれず、コンテキストを節約する
- 対応する `*.opt-in.md` ファイルがない拡張機能は常に強制される。ワークフロー開始時にそれらのルールファイルを直ちに読み込む

**強制(読み込み済み/有効化された拡張機能のみに適用)**:

- 拡張機能ルールはハード制約であり、任意のガイダンスではない
- 各段階で、モデルは段階の目的、生成されるアーティファクト、作業のコンテキストに基づいて、どの拡張機能ルールが適用可能かをインテリジェントに評価し、関連するルールのみを強制する
- 現在の段階に適用できないルールは、コンプライアンスサマリーで N/A としてマークされる(これはブロッキングな指摘ではない)
- 適用可能で有効化されている拡張機能ルールへの不適合は **ブロッキングな指摘** である。解決されるまで段階完了を提示しないこと
- 段階完了を提示する際は、拡張機能ルールのコンプライアンスサマリーを含める(ルールごとに適合/不適合/N/A と、N/A 判定の簡単な根拠)

**条件付き強制**: 拡張機能は条件付きで有効/無効にできます。オプトインメカニズムについては `inception/requirements-analysis.md` を参照してください。どの段階でも拡張機能を強制する前に、`aidlc-docs/aidlc-state.md` の `## Extension Configuration` セクションで `Enabled` ステータスをチェックしてください。無効な拡張機能はスキップし、スキップを audit.md に記録してください。設定が存在しない場合はデフォルトで強制します。

## 必須: コンテンツ検証

**重要**: いかなるファイルを作成する前にも、`common/content-validation.md` のルールに従ってコンテンツを検証しなければなりません:

- Mermaid ダイアグラム構文を検証する
- ASCII アートダイアグラムを検証する(`common/ascii-diagram-standards.md` 参照)
- 特殊文字を適切にエスケープする
- 複雑な視覚的コンテンツにはテキストの代替を提供する
- コンテンツのパース互換性をテストする

## 必須: 質問ファイルフォーマット

**重要**: どのフェーズでも質問をする際は、質問フォーマットガイドラインに従わなければなりません。

**完全な質問フォーマットルールについては `common/question-format-guide.md` を参照してください。以下を含みます**:

- 多肢選択式フォーマット(A、B、C、D、E の選択肢)
- [Answer]: タグの使用法
- 回答の検証と曖昧さの解決

## 必須: カスタムウェルカムメッセージ

**重要**: ソフトウェア開発リクエストを開始する際は、常にウェルカムメッセージを表示しなければなりません。

**ウェルカムメッセージの表示方法**:

1. 解決されたルール詳細ディレクトリの `common/welcome-message.md` からウェルカムメッセージを読み込む
2. 完全なメッセージをユーザーに表示する
3. これは新しいワークフローの開始時に一度だけ行うべきである
4. コンテキストスペースを節約するため、以降のやり取りではこのファイルを読み込まないこと

# インセプションフェーズ(INCEPTION PHASE)

**目的**: 計画、要件収集、およびアーキテクチャ上の決定

**焦点**: 何を(WHAT)、なぜ(WHY)構築するかを決定する

**インセプションフェーズの段階**:

- ワークスペース検出(常に実行)
- リバースエンジニアリング(条件付き - ブラウンフィールドのみ)
- 要件分析(常に実行 - 適応的な深度)
- ユーザーストーリー(条件付き)
- ワークフロー計画(常に実行)
- アプリケーション設計(条件付き)
- ユニット生成(条件付き)

---

## ワークスペース検出(常に実行)

1. **必須**: 最初のユーザーリクエストを完全な生の入力のまま audit.md に記録する
2. `inception/workspace-detection.md` からすべてのステップを読み込む
3. ワークスペース検出を実行する:
   - 既存の aidlc-state.md をチェックする(見つかった場合は再開)
   - ワークスペースの既存コードをスキャンする
   - ブラウンフィールドかグリーンフィールドかを判断する
   - 既存のリバースエンジニアリング成果物をチェックする
4. 次のフェーズを決定する: リバースエンジニアリング(ブラウンフィールドかつ成果物がない場合)または要件分析
5. **必須**: 発見内容を audit.md に記録する
6. 完了メッセージをユーザーに提示する(メッセージフォーマットは workspace-detection.md 参照)
7. 自動的に次のフェーズへ進む

## リバースエンジニアリング(条件付き - ブラウンフィールドのみ)

**以下の場合に実行**:

- 既存のコードベースが検出された
- 以前のリバースエンジニアリング成果物が見つからない

**以下の場合にスキップ**:

- グリーンフィールドプロジェクト
- 以前のリバースエンジニアリング成果物が存在する

**実行**:

1. **必須**: リバースエンジニアリングの開始を audit.md に記録する
2. `inception/reverse-engineering.md` からすべてのステップを読み込む
3. リバースエンジニアリングを実行する:
   - すべてのパッケージとコンポーネントを分析する
   - システム全体のビジネストランザクションを網羅するビジネス概要を生成する
   - アーキテクチャドキュメントを生成する
   - コード構造ドキュメントを生成する
   - API ドキュメントを生成する
   - コンポーネントインベントリを生成する
   - ビジネストランザクションがコンポーネント間でどのように実装されているかを示すインタラクション図を生成する
   - 技術スタックドキュメントを生成する
   - 依存関係ドキュメントを生成する

4. **明示的な承認を待つ**: 詳細な完了メッセージを提示する(メッセージフォーマットは reverse-engineering.md 参照)- ユーザーが確認するまで進まないこと
5. **必須**: ユーザーの応答を完全な生の入力のまま audit.md に記録する

## 要件分析(常に実行 - 適応的な深度)

**常に実行されます**が、リクエストの明確さと複雑さによって深度が変わります:

- **最小**: シンプルで明確なリクエスト - 意図分析を文書化するだけ
- **標準**: 通常の複雑さ - 機能要件と非機能要件を収集する
- **包括的**: 複雑でリスクが高い - トレーサビリティ付きの詳細要件

**実行**:

1. **必須**: このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/requirements-analysis.md` からすべてのステップを読み込む
3. 要件分析を実行する:
   - リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
   - ユーザーリクエストを分析する(意図分析)
   - 必要な要件の深度を判断する
   - 現在の要件を評価する
   - 必要に応じて明確化のための質問をする
   - 要件ドキュメントを生成する
4. 適切な深度(最小/標準/包括的)で実行する
5. **明示的な承認を待つ**: requirements-analysis.md の詳細ステップに従った承認フォーマットを使用する - ユーザーが確認するまで進まないこと
6. **必須**: ユーザーの応答を完全な生の入力のまま audit.md に記録する

## ユーザーストーリー(条件付き)

**インテリジェントな評価**: ユーザーストーリーが価値を追加するかどうかを、多要素分析を用いて判断します:

**常に実行する場合**(高優先度インジケータ):

- 新しいユーザー向け機能や機能性
- ユーザーワークフローや操作に影響する変更
- 複数のユーザータイプやペルソナが関与する
- 受け入れ基準が必要な複雑なビジネス要件
- クロスファンクショナルなチームコラボレーションが必要
- 顧客向け API またはサービスの変更
- 新しい製品機能や拡張

**実行する可能性が高い場合**(中優先度 - 複雑さを評価):

- 既存のユーザー向け機能の変更
- ユーザー体験に間接的に影響するバックエンド変更
- ユーザーワークフローに影響する統合作業
- ユーザーに見えるパフォーマンス改善
- ユーザー操作に影響するセキュリティ強化
- ユーザーデータまたはレポートに影響するデータモデル変更

**複雑さベースの評価**: 中優先度のケースでは、以下の場合にユーザーストーリーを実行します:

- リクエストが複数のコンポーネントやサービスに関わる
- 変更が複数のユーザータッチポイントにまたがる
- ビジネスロジックが複雑または複数のシナリオがある
- 要件に曖昧さがあり、ストーリーが明確化に役立つ
- 実装が複数のユーザージャーニーに影響する
- 変更がビジネスに重大な影響やリスクを与える

**スキップする場合のみ**(低優先度 - シンプルなケース):

- ユーザーへの影響が全くない純粋な内部リファクタリング
- 明確で限定的なスコープのシンプルなバグ修正
- ユーザーに見える影響のないインフラ変更
- 機能変更のない技術的負債のクリーンアップ
- 開発者ツールやビルドプロセスの改善
- ドキュメントのみの更新

**評価基準**: 迷った場合は、以下のケースではユーザーストーリーの含有を優先します:

- ビジネスステークホルダーが関与するリクエスト
- ユーザー受け入れテストを必要とする変更
- 複数の実装アプローチがある機能
- チーム共通理解から利益を得られる作業
- 要件の明確さが価値のあるプロジェクト

**評価プロセス**:

1. リクエストの複雑さとスコープを分析する
2. ユーザーへの影響(直接的または間接的)を特定する
3. ビジネスコンテキストとステークホルダーのニーズを評価する
4. チームコラボレーションの利益を考慮する
5. ボーダーラインケースでは含有をデフォルトとする

**注**: 要件分析が実行された場合、ストーリーはそれらの要件を参照し、それに基づいて構築できます。

**ユーザーストーリーは 1 つの段階内に 2 つのパートがあります**:

1. **パート 1 - 計画**: 質問付きのストーリー計画を作成し、回答を収集し、曖昧さを分析し、承認を得る
2. **パート 2 - 生成**: 承認された計画を実行してストーリーとペルソナを生成する

**実行**:

1. **必須**: このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/user-stories.md` からすべてのステップを読み込む
3. **必須**: インテリジェントな評価(user-stories.md のステップ 1)を実行し、ユーザーストーリーが必要であることを検証する
4. リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
5. 要件が存在する場合、ストーリーを作成する際にそれらを参照する
6. 適切な深度(最小/標準/包括的)で実行する
7. **パート 1 - 計画**: 質問付きのストーリー計画を作成し、ユーザーの回答を待ち、曖昧さを分析し、承認を得る
8. **パート 2 - 生成**: 承認された計画を実行してストーリーとペルソナを生成する
9. **明示的な承認を待つ**: user-stories.md の詳細ステップに従った承認フォーマットを使用する - ユーザーが確認するまで進まないこと
10. **必須**: ユーザーの応答を完全な生の入力のまま audit.md に記録する

## ワークフロー計画(常に実行)

1. **必須**: このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/workflow-planning.md` からすべてのステップを読み込む
3. **必須**: `common/content-validation.md` からコンテンツ検証ルールを読み込む
4. すべての事前コンテキストを読み込む:
   - リバースエンジニアリング成果物(ブラウンフィールドの場合)
   - 意図分析
   - 要件(実行された場合)
   - ユーザーストーリー(実行された場合)
5. ワークフロー計画を実行する:
   - どのフェーズを実行するかを決定する
   - 各フェーズの深度レベルを決定する
   - マルチパッケージ変更シーケンスを作成する(ブラウンフィールドの場合)
   - ワークフロービジュアライゼーションを生成する(書き込み前に Mermaid 構文を検証する)
6. **必須**: ファイル作成前にすべてのコンテンツを content-validation.md ルールに従って検証する
7. **明示的な承認を待つ**: workflow-planning.md ステップ 9 の言葉遣いを使って推奨事項を提示し、ユーザーが推奨を上書きする権限を強調する - ユーザーが確認するまで進まないこと
8. **必須**: ユーザーの応答を完全な生の入力のまま audit.md に記録する

## アプリケーション設計(条件付き)

**以下の場合に実行**:

- 新しいコンポーネントまたはサービスが必要
- コンポーネントのメソッドとビジネスルールの定義が必要
- サービス層の設計が必要
- コンポーネントの依存関係の明確化が必要

**以下の場合にスキップ**:

- 既存のコンポーネント境界内の変更
- 新しいコンポーネントやメソッドがない
- 純粋な実装変更

**実行**:

1. **必須**: このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/application-design.md` からすべてのステップを読み込む
3. リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
4. 適切な深度(最小/標準/包括的)で実行する
5. **明示的な承認を待つ**: 詳細な完了メッセージを提示する(メッセージフォーマットは application-design.md 参照)- ユーザーが確認するまで進まないこと
6. **必須**: ユーザーの応答を完全な生の入力のまま audit.md に記録する

## ユニット生成(条件付き)

**以下の場合に実行**:

- システムを複数の作業単位に分解する必要がある
- 複数のサービスやモジュールが必要
- 構造化された分解を必要とする複雑なシステム

**以下の場合にスキップ**:

- 単一のシンプルなユニット
- 分解が不要
- 単純な単一コンポーネントの実装

**実行**:

1. **必須**: このフェーズ中のユーザー入力を audit.md に記録する
2. `inception/units-generation.md` からすべてのステップを読み込む
3. リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
4. 適切な深度(最小/標準/包括的)で実行する
5. **明示的な承認を待つ**: 詳細な完了メッセージを提示する(メッセージフォーマットは units-generation.md 参照)- ユーザーが確認するまで進まないこと
6. **必須**: ユーザーの応答を完全な生の入力のまま audit.md に記録する

# 🟢 コンストラクションフェーズ(CONSTRUCTION PHASE)

**目的**: 詳細設計、NFR 実装、およびコード生成

**焦点**: どのように(HOW)構築するかを決定する

**コンストラクションフェーズの段階**:

- ユニットごとのループ(各ユニットに対して実行):
  - 機能設計(条件付き、ユニットごと)
  - NFR 要件(条件付き、ユニットごと)
  - NFR 設計(条件付き、ユニットごと)
  - インフラストラクチャ設計(条件付き、ユニットごと)
  - コード生成(常に実行、ユニットごと)
- ビルドとテスト(常に実行 - すべてのユニット完了後)

**注**: 次のユニットに進む前に、各ユニットは完全に(設計+コード)完了します。

---

## ユニットごとのループ(各ユニットに対して実行)

**各作業単位について、以下の段階を順番に実行します:**

### 機能設計(条件付き、ユニットごと)

## オペレーション(プレースホルダー)

**ステータス**: この段階は現在、将来の拡張のためのプレースホルダーです。

オペレーション段階は最終的に以下を含みます:

- デプロイメントの計画と実行
- 監視と可観測性のセットアップ
- インシデント対応手順
- メンテナンスとサポートのワークフロー
- プロダクションレディネスチェックリスト

**現状**: すべてのビルドとテストアクティビティはコンストラクションフェーズで処理されます。

## 主要な原則

- **適応的実行**: 価値を追加する段階のみを実行する
- **透明な計画**: 開始前に常に実行計画を示す
- **ユーザーコントロール**: ユーザーは段階の包含/除外を要求できる
- **進捗追跡**: 実行済みおよびスキップ済みの段階で aidlc-state.md を更新する
- **完全な監査証跡**: すべてのユーザー入力と AI の応答をタイムスタンプ付きで audit.md に記録する
  - **重要**: ユーザーの完全な生の入力を提供されたとおりに正確にキャプチャする
  - **重要**: 監査ログでユーザー入力を要約または言い換えしないこと
  - **重要**: 承認だけでなく、すべてのやり取りを記録する
- **品質重視**: 複雑な変更にはフル処理を、シンプルな変更は効率的に保つ
- **コンテンツ検証**: ファイル作成前に常にコンテンツを content-validation.md ルールに従って検証する
- **発生的動作禁止**: コンストラクションフェーズは、それぞれのルールファイルで定義された標準化された 2 択完了メッセージを必ず使用すること。3 択メニューやその他の発生的なナビゲーションパターンを作成しないこと。

## 必須: プランレベルのチェックボックス強制

### プラン実行のための必須ルール

1. **プランのチェックボックスを更新せずに作業を完了しないこと**
2. **プランファイルに記述されたステップを完了した直後に、そのステップを [x] としてマークする**
3. **これは作業が完了したのと同じインタラクションで行われなければならない**
4. **例外なし**: すべてのプランステップの完了はチェックボックスの更新で追跡されなければならない

### 2 レベルのチェックボックス追跡システム

- **プランレベル**: 各段階内の詳細な実行進捗を追跡する
- **段階レベル**: aidlc-state.md の全体的なワークフロー進捗を追跡する
- **すぐに更新**: すべての進捗更新は、作業が完了したのと同じインタラクションで行う

## プロンプトログ要件

- **必須**: すべてのユーザー入力(プロンプト、質問、応答)をタイムスタンプ付きで audit.md に記録する
- **必須**: ユーザーの完全な生の入力を提供されたとおりに正確にキャプチャする(決して要約しない)
- **必須**: ユーザーに尋ねる前に、すべての承認プロンプトをタイムスタンプ付きで記録する
- **必須**: 受信後、すべてのユーザー応答をタイムスタンプ付きで記録する
- **重要**: 常に audit.md ファイルを編集して追記(APPEND)する。内容を完全に上書きするツールやコマンドは決して使用しないこと
- **重要**: audit.md の内容全体を上書きするファイル書き込みツールやコマンドは絶対に使用しないこと。重複の原因となります
- タイムスタンプには ISO 8601 形式(YYYY-MM-DDTHH:MM:SSZ)を使用する
- 各エントリに段階コンテキストを含める

### 監査ログフォーマット:

```markdown
## [段階名またはインタラクションタイプ]

**Timestamp**: [ISO タイムスタンプ]
**User Input**: "[完全な生のユーザー入力 - 決して要約しない]"
**AI Response**: "[AI の応答または取られたアクション]"
**Context**: [段階、アクション、または下された決定]

---
```

### audit.md の正しいツールの使用

✅ 正しい方法:

1. audit.md ファイルを読む
2. ファイルを追記/編集して変更を加える

❌ 間違った方法:

1. audit.md ファイルを読む
2. 読んだ内容に追加したい新しい変更を加えたものを、audit.md に完全に上書きする
